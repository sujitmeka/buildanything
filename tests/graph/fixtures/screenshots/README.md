# Slice 5 Screenshot Fixtures

These four `.png` files are **not real PNG images** — they are synthetic 1024-byte buffers generated by a deterministic LCG pseudo-random sequence. The Slice 5 screenshot extractor is in stub mode (see `src/graph/parser/screenshot.ts`): it does not decode PNG headers, it computes the perceptual hash directly over raw bytes via the stub `dhash` in `src/graph/util/dhash.ts`. Synthetic bytes are therefore sufficient to exercise extraction, hashing, and similarity queries without committing real binary screenshots.

## Generation

Each fixture is a 1024-byte LCG (Numerical Recipes constants: `s = (s * 1664525 + 1013904223) mod 2^32`, top byte taken). The `marketplace-hero-prod.png` fixture is `marketplace-hero-reference.png` with a single byte XOR-flipped at offset 504 (the byte sampled for sample index 32 in the dhash window) — this produces a perceptual hash exactly 1 bit away from the reference, simulating a small visual change between a design reference and a production capture.

If you need to regenerate these (for example, after changing the dhash algorithm), run a small script that mirrors the generator above and writes the four files in this directory. Verify by running `tests/graph/screenshot-parser.test.ts` and `tests/graph/slice5-queries.test.ts` — they assert specific hash distances.

## What each fixture exercises

| File | LCG seed | Modification | Role in tests |
|---|---|---|---|
| `marketplace-hero-reference.png` | `0x1234abcd` | none | Base design reference. Used to test reference-class extraction (component detection, dna_axis_tags inference from filename, palette emission). |
| `marketplace-hero-prod.png` | `0x1234abcd` | byte at 504 XOR `0x80` | Brand-drift candidate. Produces a dhash 1 bit away from the reference — exercises `queryScreenshotSimilar` in the close-match regime (distance ≤ 5). Also exercises the brand_drift class path in the extractor. |
| `dashboard-card-reference.png` | `0xdeadbeef` | none | Distinct reference. Hash distance ≥ 30 from marketplace fixtures — exercises the threshold cutoff in `queryScreenshotSimilar` (no false positives when bytes are unrelated). |
| `dogfood-error-1.png` | `0xfeed1234` | none | Dogfood-class fixture. Used to exercise `linkedFindingId` wiring and the `screenshot_evidences_finding` edge. |

## Why bytes, not real PNGs

The stub extractor's `image_dimensions` field is hard-coded to `"0x0"` and `dominant_palette` is a fixed two-color array — both are placeholders for the production-mode Claude Vision integration. Adding real PNG headers would not change any extraction output; it would only inflate the fixture size. The path-only contract (per Slice 5 schema §10 redline 5) means the graph never stores image bytes, so fixture portability does not require a valid PNG signature.

When Slice 5 production mode lands (real image decoding via Sharp / @napi-rs/image), these fixtures will be replaced with small real PNG files — but the test shape (assert hashes, assert similar matches, assert linking edges) stays the same.
